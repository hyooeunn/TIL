# 📍 멀티, 모너, 터보 레포

## 멀티 레포

> 프로젝트별로 레포지토리를 생성해 만든 것

**1레포의 1프로젝트**로 각 프로젝트가 고유의 저장소를 가지게 되고, 다른 프로젝트와의 의존성을 가지고 있지 않아 독립적으로 빠르게 개발이 가능하다.

### 장점

- 독립적 배포 및 개발
  - 특정 프로젝트만 선택적으로 업데이트하거나 배포하기가 쉽다.
  - 각 팀이 다른 팀 코드에 의존하지 않고 빠르게 배포 가능
- 다양한 기술 스택
  - 언어, 빌드 도구를 사용하는 프로젝트 간 충돌 방지
- 다른 프로젝트의 영향 x
  - 하지만 단점이 될 수도 있음
  - 독립적 관리로 인해 한 저장소에서 문제가 발생해도 다른 프로젝트에 영향을 주지 않음

### 단점

- 의존성 관리 복잡, 불편 -> 전부 관리하기 때문
- 공통 라이브러리, 설정 파일 여러 저장소에서 중복 관리할 때 충돌 가능성 있음
- 공통 코드 관리 어려움
  - 공통 라이브러리 설정이나 동기화가 까다로움
- 프로젝트 별로 CI / CD 설정 필요

## 모노 레포

> 여러 프로젝트를 하나의 레포지토리로 한 번에 관리

**1레포 n프로젝트**로 시스템의 각 서비스를 하나의 레포지토리에서 일괄 관리한다

### 장점

- 빠른 코드 수정
  - 같은 코드 중복 제거
  - 서버와 클라이언트가 같은 저장소에 있어, 공통된 코드를 쉽게 재사용
- 의존성 충돌 x
  - 한 저장소에서 관리되므로 의존성 버전 충돌을 줄임
- 코드 컨벤션 통일
- ⭐ CI / CD 간편
  - 한 저장소에서 빌드와 설정 과정이 단순화된다

### 단점

- 의존성 관리 복잡 (과도한 의존성)
  - 특정 의존성을 업데이트시 다른 프로젝트에 영향을 미칠 수 있음
- 코드 충돌
  - 동일한 코드를 수정하면 다른 코드에도 영향이 갈 수 있음
- CI 속도 저하
  - 대규모 프로젝트의 경우 저장소 크기가 켜져 무거워짐으로 클론, 빌드, 테스트 시간이 증가

## 터보 레포

> JS, TS 베이스의 모노레포를 위한 고성능 빌드 시스템
>
> - 개발자 부담 ⬇️

### 터보 레포의 특징

1. `Incremental builds`: 한 번 빌드된 파일을 캐싱한 뒤 다음 빌드를 수행할때까지 건너 뜀

- 빌드를 딱 한 번 하는 것을 목표

2. `Content-aware hasing`: 타임스탬프가 아닌 콘텐츠를 인식하는 방식으로 해싱 지원

- **타임스탬프**: 파일을 마지막으로 수정한 시간

3. `Cloud caching`: 빌드 결과물을 클라우드 공유 폴더에 올려놓는 것

   ```md
   팀 프로젝트에서 한 명이 빌드하면
   -> 클라우드 저장
   -> 팀원 전원 재사용
   -> CI 서버도 재사용

   결과:
   팀 전체의 빌드 시간이 극적으로 줄어듬
   ```

4. `Parallel execution`: 의존성을 판단해 최대한 병렬적으로 작업 진행

- ex) 혼자 순서대로 (직렬) 하던 일을 여러명이 동시에 나눠서 하는 것 (병렬)

5. `Task Pipelines`: 연결 정의 후 빌드를 언제 어떻게 실행할지 판단해 최적화

- ex) 이 작업은 저 작업이 끝나야 시작할 수 있어"=
  를 미리 정의해두는 것

6. `Zero Runtime Overhead`: 런타임 코드와 소스맵을 다루지 않음 따라서 런타임 단계에서 파악하지 못한 리스트가 불거질 위험 x

- **런타임**: 사용자가 앱을 실제로 사용하는 순간
- **소스 맵**: 빌드된 코드와 원본 코드를 연결하는 지도

7. `Pruned subsets`: 빌드에 필요한 요소만으로 모노 레포의 하위 집합을 생성해 배포 속도를 높임

8. `JSON configuration`: 별도 코드 작업 x, JSON 설정으로 터보 사용

9. `Profile in browser`: 빌드 프로필로 빌드 과정을 시각화 / 병목 지점을 쉽게 찾음

### 터보 레포 간단 사용법

> - 모노레포 root에 `turbo.json` 파일 생성
> - 파이프라인(pipeline) 작성
>   - 파이프라인의 각 명령어들은 하나의 테스크 단위
>   - 이 단위가 병렬 처리 및 의존성 범위가 됨

```json
{
  "baseBranch": "origin/main",
  "pipeline": {
    "build": {
      // ^는 커맨드 실행 전 dependencies 혹은 devDependencies의 위상 의존성을 가질 때 명시해 줍니다(https://turborepo.org/docs/glossary#topological-order).
      // 의존성 빌드 명령이 실행된 후 build 커맨드가 실행됩니다.
      "dependsOn": ["^build"],
      // 기본 캐시 폴더를 지정합니다.
      "outputs": [".next/**", "lib/**", "storybook-static/**"]
    },
    "cypress:ci": {
      "dependsOn": [
        // 특정 패키지를 지정하고 싶다면 '패키지명#스크립트'로 하면 됩니다.
        "@linecorp/uvp#build"
      ]
    },
    // 아무런 명시가 없다면 의존성이 없다는 것을 의미하며, 이는 언제든지 실행될 수 있다는 것을 의미합니다.
    // 작업이 가능할 때마다 실행합니다.
    "lint": {},
    "deploy": {
      // 의존성을 여러 개 지정할 경우 터보가 똑똑하게 순서를 맞춰서 진행해 줍니다. 위 'Profile in browser'의 이미지를 참고해 주세요.
      "dependsOn": ["build", "cypress:ci", "snapshots", "lint"]
    },
    // 개발 환경과 같이 핫 로딩이 필요할 경우 캐시를 비활성화할 수 있습니다.
    "dev": {
      "cache": false
    }
  }
}
```
